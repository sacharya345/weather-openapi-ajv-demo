# Weather API Demo - Code Review: Request Validation in Practice

This document walks you through the Weather API server and explains exactly how
incoming request validation works, why the default validation was disabled, and
how a custom AJV-based validation replaces it.

Read this document **with your editor open** so you can follow along in the actual
source files.

---

## 1. Project Purpose

### What this project demonstrates

This project is a working example of **contract-first API development**. Instead of
writing code first and documenting it later, we start with a formal specification
(the OpenAPI document) and let tooling generate the server skeleton from it.

The project then shows you three stages of request validation:

1. **Generated (framework) validation** - the out-of-the-box validation that
   `express-openapi-validator` provides automatically.
2. **Disabling that validation** - what happens when you turn it off and why
   you might want to.
3. **Custom validation with AJV** - building your own validation middleware that
   reads the same OpenAPI schema, giving you full control over validation logic
   and error formatting.

### What "contract-first" means

In contract-first development, the **OpenAPI specification is the single source of
truth**. Everything flows from it:

```
openapi.yaml (the contract)
    |
    +--> Code generation (server skeleton)
    +--> Request validation rules
    +--> API documentation (Swagger UI)
    +--> Client SDKs (if needed)
```

The specification defines *what* the API accepts and returns. The code implements
*how* it behaves. This separation is the core idea.

## 2. High-Level Architecture

### Express server structure

The server follows the standard structure generated by **OpenAPI Generator** using
the `nodejs-express-server` template. Here is the file layout with the role of each
file:

```
weather-server/
  index.js                  <-- Entry point: creates and launches the server
  expressServer.js          <-- Server setup: middleware chain, validation, routing
  config.js                 <-- Configuration constants (port, paths)
  logger.js                 <-- Winston logger setup
  api/
    openapi.yaml            <-- THE CONTRACT: OpenAPI 3.0 specification
  controllers/
    Controller.js           <-- Base controller: request handling logic
    DefaultController.js    <-- Generated controller: routes to service functions
    index.js                <-- Controller exports
  services/
    Service.js              <-- Base service: response formatting helpers
    DefaultService.js       <-- Generated service: business logic goes here
    index.js                <-- Service exports
  utils/
    openapiRouter.js        <-- OpenAPI-aware routing middleware
```

### Role of the OpenAPI spec

The file `api/openapi.yaml` defines:

- **Paths**: `/weather` (GET and POST)
- **Schemas**: The `Weather` object with its required fields and types
- **Operation handlers**: Via `x-eov-operation-handler` annotations that tell the
  framework which controller handles each operation

Here is the Weather schema from the spec:

```yaml
# api/openapi.yaml - lines 41-70
Weather:
  additionalProperties: false
  properties:
    sensorId:
      description: Unique identifier for the sensor
      type: integer
    cityCode:
      description: City identifier code
      type: string
    temperature:
      description: Temperature reading in Celsius
      type: number
    windSpeed:
      description: Wind speed in m/s
      type: number
  required:
    - cityCode
    - sensorId
    - temperature
    - windSpeed
  type: object
```

Key observations:
- `additionalProperties: false` - no extra fields allowed
- All four fields are `required`
- Types are strictly defined (`integer`, `string`, `number`)

### Role of OpenAPI Generator

OpenAPI Generator CLI read `openapi.yaml` and produced:
- The controller files (`DefaultController.js`)
- The service files (`DefaultService.js`)
- The routing utility (`openapiRouter.js`)
- The Express server skeleton (`expressServer.js`)
- The base classes (`Controller.js`, `Service.js`)

You should recognize that **most of this code was generated, not hand-written**.
The files you see under `controllers/`, `services/`, and `utils/` came from the
generator. The modifications made by the developer are concentrated in
`expressServer.js`.

### Main files students should inspect

Focus your attention on these files, in this order:

| Priority | File | Why |
|----------|------|-----|
| 1 | `api/openapi.yaml` | The contract - everything starts here |
| 2 | `expressServer.js` | Middleware chain, AJV validation, framework config |
| 3 | `controllers/DefaultController.js` | How requests reach the service layer |
| 4 | `services/DefaultService.js` | Where business logic lives |
| 5 | `controllers/Controller.js` | How request parameters are extracted |

## 3. Default Validation (Generated)

### Where express-openapi-validator is initialized

In `expressServer.js`, lines 103-111, the `express-openapi-validator` middleware is
registered:

```javascript
// expressServer.js - lines 103-111
this.app.use(
  OpenApiValidator.middleware({
    apiSpec: this.openApiPath,
    operationHandlers: path.join(__dirname),
    fileUploader: { dest: config.FILE_UPLOAD_PATH },

    validateRequests: false    // <-- THIS IS THE KEY CHANGE
  }),
);
```

In a freshly generated project, the line `validateRequests: false` would either be
absent or set to `true`. The default behavior of `express-openapi-validator` is to
**validate every incoming request** against the OpenAPI spec automatically.

### How it reads the OpenAPI spec

When `OpenApiValidator.middleware()` is called, it:

1. Loads and parses the `apiSpec` file (`api/openapi.yaml`)
2. Builds an internal model of all paths, methods, parameters, and schemas
3. Creates Express middleware functions that intercept every request
4. Matches each request to its path definition in the spec
5. Validates request body, query parameters, path parameters, and headers

### How request validation normally works (when enabled)

With `validateRequests: true` (the default), the middleware chain works like this:

```
Incoming POST /weather request
    |
    v
[body-parser] -- parses JSON body
    |
    v
[express-openapi-validator] -- checks:
    |   - Is the Content-Type correct?
    |   - Does the body match the Weather schema?
    |   - Are all required fields present?
    |   - Are field types correct?
    |   - Are there extra fields? (additionalProperties: false)
    |
    +-- FAIL --> 400 error response (automatic)
    |
    +-- PASS --> continues to controller --> service
```

### What kind of errors it produces

The framework produces error responses in its own format. For example, sending a
request missing `cityCode` would produce something like:

```json
{
  "message": "request/body must have required property 'cityCode'",
  "errors": [
    {
      "path": "/body/cityCode",
      "message": "must have required property 'cityCode'",
      "errorCode": "required.openapi.validation"
    }
  ]
}
```

This format is **defined by the framework**, not by you. You have limited control
over how these error messages look or what information they contain.

## 4. Disabling Default Validation

### What `validateRequests: false` does

```javascript
// expressServer.js - line 109
validateRequests: false
```

This single line tells `express-openapi-validator` to **skip all request body
validation**. The middleware still runs, but it no longer checks whether incoming
data matches the schema.

### Why the server still works

Disabling validation does **not** disable routing. The `express-openapi-validator`
middleware has two separate responsibilities:

| Responsibility | Still active? |
|----------------|---------------|
| **Routing** - matching requests to `operationHandlers` | Yes |
| **Request validation** - checking body/params against schema | No |

The `operationHandlers` option tells the framework to use the
`x-eov-operation-handler` annotations in the spec to find the right controller:

```yaml
# api/openapi.yaml - line 38
x-eov-operation-handler: controllers/DefaultController
```

This routing mechanism still works. Requests still reach `DefaultController.js`,
which calls `Controller.handleRequest()`, which calls the service function.

### Why invalid payloads now pass through

With validation disabled, any JSON body will be accepted by the framework layer.
You could send:

```json
{ "nonsense": true }
```

or:

```json
{ "cityCode": 12345, "temperature": "hot" }
```

Both would pass through the framework middleware and arrive at the service layer.
There is **nothing stopping invalid data** at the framework level.

This is exactly the problem that the custom AJV validation solves.

### What remains active

- **Routing**: Requests are still matched to the correct controller and operation
- **Operation handlers**: `x-eov-operation-handler` still maps paths to controllers
- **Path matching**: Unknown paths still produce 404s
- **Swagger UI**: The `/api-docs` endpoint still serves interactive documentation
- **Body parsing**: `body-parser` and `express.json()` still parse JSON

## 5. Custom AJV Validation

### What AJV is

**AJV (Another JSON Validator)** is a standalone JSON Schema validation library.
It is the fastest JSON Schema validator for JavaScript and can validate data against
any JSON Schema - including schemas extracted from an OpenAPI document.

The key packages used:
- `ajv` (v8) - the core validator
- `ajv-formats` - adds support for format validation (e.g., `date-time`, `email`)

```javascript
// expressServer.js - lines 2-3
const Ajv = require('ajv');
const addFormats = require('ajv-formats');
```

### How it loads the schema from OpenAPI

The developer wrote two helper functions at the top of `expressServer.js`:

```javascript
// expressServer.js - lines 18-27
function loadOpenApiSpec(openApiYamlPath) {
  const raw = fs.readFileSync(openApiYamlPath, 'utf8');
  return jsYaml.safeLoad(raw);
}

function getWeatherSchema(spec) {
  const schema = spec?.components?.schemas?.Weather;
  if (!schema) throw new Error("Weather schema not found");
  return schema;
}
```

`loadOpenApiSpec()` reads and parses the YAML file. `getWeatherSchema()` navigates
into the parsed spec object and extracts the `Weather` schema from
`components.schemas.Weather`.

This is important: **the AJV validator uses the exact same schema that the OpenAPI
spec defines**. The contract remains the single source of truth.

### How the Weather schema is extracted

Inside `setupMiddleware()`, these functions are called:

```javascript
// expressServer.js - lines 52-53
const spec = loadOpenApiSpec(this.openApiPath);
const weatherSchema = getWeatherSchema(spec);
```

The resulting `weatherSchema` object is the JavaScript equivalent of the YAML
schema definition:

```javascript
{
  additionalProperties: false,
  properties: {
    sensorId:    { description: "...", type: "integer" },
    cityCode:    { description: "...", type: "string" },
    temperature: { description: "...", type: "number" },
    windSpeed:   { description: "...", type: "number" }
  },
  required: ["cityCode", "sensorId", "temperature", "windSpeed"],
  type: "object"
}
```

### How `ajv.compile` works

```javascript
// expressServer.js - lines 55-58
const ajv = new Ajv({ allErrors: true, strict: false });
addFormats(ajv);

const validateWeather = ajv.compile(weatherSchema);
```

Step by step:

1. `new Ajv({ allErrors: true, strict: false })` creates an AJV instance.
   - `allErrors: true` - reports **all** validation errors, not just the first one
   - `strict: false` - allows OpenAPI-specific keywords like `example` and
     `description` that are not part of standard JSON Schema

2. `addFormats(ajv)` registers format validators (date, email, uri, etc.)

3. `ajv.compile(weatherSchema)` takes the schema and returns a **validation
   function** (`validateWeather`). This function is pre-compiled for performance -
   the schema is analyzed once at startup, not on every request.

The result, `validateWeather`, is a function you call like this:
```javascript
const isValid = validateWeather(someData);
// isValid === true  --> data matches the schema
// isValid === false --> data does not match; errors in validateWeather.errors
```

### How the middleware is attached

```javascript
// expressServer.js - lines 60-86
const weatherValidateMw = (req, res, next) => {
  const ok = validateWeather(req.body);

  // Business rule: temperature range check
  const t = req.body?.temperature;
  if (ok && typeof t === 'number' && (t < -60 || t > 60)) {
    return res.status(400).json({
      error: "VALIDATION_FAILED",
      message: "temperature must be between -60 and 60",
      details: [{ path: "/temperature", message: "out of allowed range" }]
    });
  }

  // Schema validation failure
  if (!ok) {
    return res.status(400).json({
      error: "VALIDATION_FAILED",
      message: "Invalid weather payload",
      details: (validateWeather.errors || []).map(e => ({
        path: e.instancePath || e.schemaPath,
        message: e.message
      }))
    });
  }

  next();   // <-- all checks passed, continue to next middleware
};

this.app.post('/weather', weatherValidateMw);
```

The middleware function does three things:

1. **Schema validation**: Calls `validateWeather(req.body)` to check if the body
   matches the Weather schema
2. **Business rule validation**: If schema validation passes, checks whether
   `temperature` is within the range -60 to 60 (a rule that cannot be expressed
   in the OpenAPI schema as generated)
3. **Error or continue**: Returns a 400 error with a custom format if validation
   fails, or calls `next()` to pass control to the next middleware

Notice the error response format: you define `error`, `message`, and `details`
exactly how you want them. This is your format, not the framework's.

### How middleware order affects execution

**This is critical to understand.** Express executes middleware in the order it is
registered. Here is the actual registration order in `expressServer.js`:

```
Line 45:  app.use(cors())                           -- 1. CORS headers
Line 46:  app.use(bodyParser.json(...))              -- 2. Parse JSON body
Line 47:  app.use(express.json())                    -- 3. Parse JSON (redundant)
Line 48:  app.use(express.urlencoded(...))           -- 4. Parse URL-encoded
Line 49:  app.use(cookieParser())                    -- 5. Parse cookies
Line 86:  app.post('/weather', weatherValidateMw)    -- 6. AJV VALIDATION (POST /weather only)
Line 90:  app.get('/hello', ...)                     -- 7. Hello endpoint
Line 92:  app.get('/openapi', ...)                   -- 8. OpenAPI spec endpoint
Line 94:  app.use('/api-docs', ...)                  -- 9. Swagger UI
Line 103: app.use(OpenApiValidator.middleware(...))   -- 10. Framework routing (validation OFF)
```

For a `POST /weather` request, the execution flow is:

```
POST /weather arrives
    |
    v
[1] cors()                  -- adds CORS headers
    |
    v
[2] bodyParser.json()       -- parses JSON body into req.body
    |
    v
[6] weatherValidateMw       -- YOUR AJV validation runs HERE
    |
    +-- FAIL --> 400 response (request stops here)
    |
    +-- PASS --> continues...
    |
    v
[10] OpenApiValidator        -- routes to DefaultController (no validation)
    |
    v
DefaultController.weatherPOST()
    |
    v
Controller.handleRequest()
    |
    v
DefaultService.weatherPOST()  -- business logic
    |
    v
Response sent
```

**Key insight**: The AJV middleware on line 86 runs **before** the
`express-openapi-validator` middleware on line 103. This is why it works as a
replacement: invalid requests are rejected before they ever reach the framework
routing layer.

Also note that `weatherValidateMw` is registered only for `POST /weather`. A
`GET /weather` request skips it entirely and goes straight to the framework router.

### How custom business rules are layered on top

Look at the temperature range check in the middleware:

```javascript
// expressServer.js - lines 63-70
const t = req.body?.temperature;
if (ok && typeof t === 'number' && (t < -60 || t > 60)) {
  return res.status(400).json({
    error: "VALIDATION_FAILED",
    message: "temperature must be between -60 and 60",
    details: [{ path: "/temperature", message: "out of allowed range" }]
  });
}
```

This check runs **only if schema validation passed** (`ok === true`). This creates
a two-layer validation approach:

```
Layer 1: Schema validation (AJV)
    - Are all required fields present?
    - Are the types correct?
    - Are there unexpected extra fields?
    |
    v (only if Layer 1 passes)
Layer 2: Business rule validation (custom code)
    - Is the temperature within a reasonable range?
    - (You could add more rules here)
```

This layering is a pattern you will see in production systems: schema validation
catches structural problems, and business rule validation catches logical problems
that schemas cannot express.

## 6. Comparison: Framework Validation vs Custom AJV

| Aspect | Framework Validation (`express-openapi-validator`) | Custom AJV Validation |
|--------|----------------------------------------------------|-----------------------|
| **Setup** | Automatic - comes with the generator | Manual - you write the middleware |
| **Control over error format** | Limited - the framework decides the response structure | Full - you define exactly what the error response looks like |
| **Business rule support** | None - only validates against the schema as-is | Yes - you can add any custom checks after schema validation |
| **Dependency on generator** | Tightly coupled - if you change generators, validation changes | Independent - works with any Express app, regardless of how it was generated |
| **Reusability** | Tied to the OpenAPI middleware pipeline | The AJV validate function can be used anywhere: middleware, services, tests, scripts |
| **Schema source** | Reads the OpenAPI spec directly | Also reads the OpenAPI spec - same source of truth |
| **When validation runs** | Inside the framework middleware (after routing setup) | Before the framework middleware - you control the order |
| **Multiple schemas** | Validates all endpoints automatically | You must set up validation per endpoint manually |
| **Performance** | Validates on every request (no pre-compilation visible to you) | Schema is compiled once at startup (`ajv.compile`), reused on every request |
| **Debugging** | Errors come from the framework internals | Your code, your errors, your breakpoints |

### When to use which

- **Framework validation** is convenient for rapid prototyping and when the
  generated error format is acceptable.
- **Custom AJV validation** is preferred when you need custom error messages,
  business rule validation, or when you want validation logic that is independent
  of any particular framework.

In production systems, custom validation is almost always preferred because you
need control over error responses and business logic.

## 7. Key Learning Outcomes

After reading this document and studying the code, you should understand:

1. **Contract-first development**: The OpenAPI specification comes first. Code is
   generated from it, not the other way around. The spec is the single source of
   truth for API structure, types, and validation rules.

2. **Middleware order matters**: Express executes middleware in registration order.
   The AJV validation middleware is registered *before* the framework routing
   middleware, which is why it can intercept and reject invalid requests before
   they reach the controller.

3. **Framework validation is convenient but inflexible**: The `express-openapi-validator`
   provides automatic validation, but you cannot control error formats or add
   business rules. Setting `validateRequests: false` disables it.

4. **Disabling validation does not break routing**: The OpenAPI middleware handles
   both routing and validation. These are independent concerns - you can disable
   one without affecting the other.

5. **AJV gives you full control**: By extracting the schema from the same OpenAPI
   spec and compiling it with AJV, you get a standalone validation function that
   you own. You decide when it runs, what errors look like, and what additional
   checks to perform.

6. **Layered validation is a pattern**: Schema validation catches structural
   errors (missing fields, wrong types). Business rules catch logical errors
   (temperature out of range). These are separate concerns and should be layered,
   not mixed.

7. **The schema is reused, not duplicated**: Both the framework validator and the
   custom AJV validator read from `api/openapi.yaml`. There is no duplication of
   validation rules - just different engines reading the same contract.

8. **Validation is middleware**: In Express, validation is just another middleware
   function. It receives `(req, res, next)`, does its work, and either sends an
   error response or calls `next()` to continue. Understanding this pattern is
   essential for building any Express application.
